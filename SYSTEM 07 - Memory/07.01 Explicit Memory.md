## Explicit Memory Definition

Explicit memory is documented, codified knowledge that can be easily retrieved and shared. It lives in documents, databases, and systems rather than in people's minds.

**Core Principle**: If it's not documented, it doesn't exist for the organization. Document important knowledge systematically.

## Types of Explicit Memory

### Process Documentation

**How work is done**:
- Process flowcharts
- Step-by-step procedures
- Decision points and criteria
- Tools and systems used
- Quality standards
- Exception handling
- Responsible parties

**Purpose**:
- Consistent execution
- Faster onboarding
- Continuous improvement
- Quality assurance
- Compliance

**Maintenance**:
- Updated when process changes
- Version controlled
- Accessible to those who use it
- Regular review (annual minimum)

### Decision Records

**Why decisions were made**:
- Decision and context
- Alternatives considered
- Rationale for choice
- Key assumptions
- Expected outcomes
- Decision date and maker
- Review status

**Purpose**:
- Understand reasoning
- Avoid re-deciding
- Learn from outcomes
- Consistency in similar decisions

**When to document**:
- Major decisions
- Decisions with significant impact
- Decisions likely to be revisited
- Decisions with complex rationale

### Project Documentation

**What was accomplished and learned**:
- Project overview and objectives
- Key milestones and timeline
- Who was involved
- What was built or accomplished
- Problems encountered and solutions
- Lessons learned
- Deliverables and handoff
- Success metrics and results

**Purpose**:
- Know what was done
- Don't redo work
- Learn from experience
- Improve future projects
- Know who has expertise

**Maintenance**:
- Close-out documentation while fresh
- Accessible archive
- Searchable by topic, client, date
- Historical reference

### Strategic Documentation

**Strategic thinking and analysis**:
- Market analysis and insights
- Competitive assessment
- Strategic plans and priorities
- Capability assessments
- Risk analyses
- Strategic initiatives
- Long-term vision

**Purpose**:
- Understand market and position
- Consistency in strategy
- Build on previous analysis
- Avoid repeating analysis
- Strategic continuity

**Review rhythm**:
- Annual update (at minimum)
- Regular board/leadership review
- Readily accessible to decision-makers

### Policy and Standards Documentation

**How we do things**:
- Organizational policies
- Quality standards
- Ethical standards
- Decision authorities
- Approval processes
- Communication protocols
- Compliance requirements

**Purpose**:
- Consistent behavior
- Clear expectations
- Compliance
- Culture codification

**Maintenance**:
- Clear ownership
- Regular review and update
- Accessible and known
- Training on key policies

### Lessons Learned Documentation

**What we learned from experience**:
- Problem encountered
- How it was solved
- What worked well
- What could have been better
- Key insight or principle
- Recommendation for future
- Date and context

**Purpose**:
- Learn from experience
- Avoid repeating mistakes
- Improve continuously
- Build organizational wisdom
- Prevent problems

**When to capture**:
- After project completion
- After significant problem resolution
- After major decision outcomes become clear
- After systematic improvement implementation
- Periodically (quarterly or annual)

### Templates and Standards

**Standardized formats**:
- Project templates
- Proposal templates
- Decision templates
- Meeting note templates
- Review templates
- Process templates
- Report templates

**Purpose**:
- Consistency
- Faster work (don't start from scratch)
- Known quality standards
- Easier integration
- Knowledge capture

**Maintenance**:
- Keep templates current
- Make easily accessible
- Version control
- Document reasons for templates

## Building Memory Repository

### What to Document

**Document**:
- Processes that are repeated
- Decisions with significant impact
- Knowledge unique to organization
- Solutions to difficult problems
- Strategic direction and rationale
- Lessons from experience
- Policies and standards

**Don't document**:
- Routine information easily found elsewhere
- Outdated content
- Superseded decisions
- One-time trivial information

### Documentation Standards

**For consistent, usable documentation**:
- Clear title and purpose
- Current date and owner
- Search tags or categories
- Executive summary for long documents
- Clear language (avoid jargon)
- Visual aids where helpful (diagrams, charts)
- Links to related documents
- Version history for changes
- Review date and who reviewed

**Format**:
- Consistent formatting (template)
- Searchable format (markdown, wiki, database)
- Mobile-accessible
- Printable if needed

### Organization Strategy

**Organizing memory for discovery**:
- By function or department
- By process type
- By decision type
- By problem category
- By time period (recent decisions, historical)
- By status (active, archived)

**Discovery tools**:
- Searchable database or wiki
- Clear taxonomy and tagging
- Index or table of contents
- Cross-references
- Related items suggestions

### Repository System

**Where memory is stored**:
- Centralized location (wiki, knowledge base, intranet)
- Version control for content
- Access control (who can see/edit)
- Backup and redundancy
- Migration strategy for platform changes
- Retirement/archival process

**Access**:
- Everyone can read relevant memory
- Clear ownership for each document
- Approval process for official policy
- Change history visible

## Capturing Explicit Memory

### Documentation Process

**When to document**:
- Real-time: During significant work (meeting notes, decisions)
- Close-out: Immediately after project completion (lessons learned)
- Routine: Regular updates (process changes, new learning)
- Retrospective: Periodic review (what have we learned?)

**Who documents**:
- Owners: Primary creator of memory
- Reviewers: Ensure accuracy and completeness
- Approvers: Validate for official documentation

**Effort**:
- Real-time documentation: Brief (5-10 minutes per event)
- Close-out documentation: Moderate (1-2 hours per project)
- Formal documentation: Significant (4-8 hours per policy/process)

### Quality of Documentation

**Good documentation**:
- Accurate (reflects reality)
- Complete (full picture)
- Clear (easy to understand)
- Current (up-to-date)
- Organized (easy to find)
- Actionable (can use it)

**Review checklist**:
- Is it accurate? (has expert reviewed?)
- Is it complete? (all relevant detail?)
- Is it clear? (understandable to audience?)
- Is it organized? (easy to find needed information?)
- Is it current? (recent date and review?)

### Documentation Maintenance

**Keeping documentation current**:
- Owner assigned to each document
- Regular review cycle (annual minimum)
- Update date clearly marked
- Version history maintained
- Outdated content archived/deleted
- Links updated when moved

**Retirement**:
- Archive when no longer relevant
- Maintain for historical reference if valuable
- Don't delete without backup
- Document why retired
- Suggest replacement if applicable

## Learning from Explicit Memory

### Using Documentation for Problem-Solving

**When facing problem**:
1. Search for similar problem in memory
2. Understand how it was solved before
3. Adapt solution to current context
4. Document outcome if different
5. Update documentation if approach improved

**Benefits**:
- Faster solution (don't start from scratch)
- Better solution (benefit from experience)
- Consistency across organization
- Continuous improvement

### Decision-Making with Memory

**When making decision**:
1. Find relevant precedents or policies
2. Understand rationale of past decisions
3. Check current context differences
4. Apply judgment to current situation
5. Document decision and reasoning

**Benefits**:
- More informed decisions
- Consistency in approach
- Reduced decision time
- Better outcomes

### Onboarding with Documentation

**New employees**:
- Access to process documentation (how we do things)
- Access to policy documentation (expectations)
- Access to strategic documentation (where we're going)
- Access to project history (what we've done)
- Access to lessons learned (what we've learned)

**Benefits**:
- Faster ramp-up (don't repeat learning curve)
- Consistent understanding
- Reduced errors
- Stronger culture transfer

## Documentation in Practice

### Meeting Minutes and Notes

**Capturing meeting memory**:
- Date, attendees, topic
- Decisions made and rationale
- Action items (what, who, when)
- Key discussion points
- Next steps
- Follow-up needed

**Distribution**:
- Send to attendees promptly
- Make available to broader team
- File in accessible location
- Update if new information emerges

### Project Close-Out

**Capturing project learning**:
- What was accomplished
- What worked well (do more of this)
- What could improve (do differently next time)
- Key problems and solutions
- Budget and schedule performance
- Recommendations for similar projects
- Handoff and transition status

**Timeline**:
- Close-out meeting while fresh (1 week after completion)
- Documentation written (2 weeks after)
- Review by leadership (1 month)
- File in accessible location

### Retrospectives

**Systematic learning**:
- Regular retrospectives (monthly, quarterly, annual)
- Review performance metrics
- Discuss what worked and what didn't
- Identify patterns and trends
- Document insights and lessons
- Plan improvements
- Track outcomes of past improvements

**Participation**:
- Team members who did the work
- Leadership sponsor
- Facilitator (impartial)
- Documentation
- Action follow-up
